## 2. 产品背景

用户在计划旅行时，通常需要同时查询景点、美食、交通、预算、天气、住宿节奏等信息。传统方式需要在多个平台之间切换，整理成本较高。

本项目希望通过大模型能力和本地知识库检索，帮助用户快速生成一份结构化旅行计划。

当前版本聚焦四川旅行场景，支持用户输入基础旅行需求，系统自动生成每日行程、预算分析和知识库引用来源。

---

## 3. 产品目标

### 3.1 核心目标

1. 用户可以通过表单输入旅行需求。
2. 系统可以生成结构化旅行计划。
3. 旅行计划按天展示，并包含上午、下午、晚上和交通建议。
4. 系统展示预算分析结果，帮助用户判断预算是否合理。
5. 系统展示 RAG 引用来源，提升结果可信度。
6. 前端具备基础加载、错误、空状态和移动端适配能力。

### 3.2 非目标

第一版不做以下功能：

1. 用户登录注册。
2. 在线支付。
3. 真实酒店、机票、门票下单。
4. 地图导航。
5. 多用户协作。
6. 历史行程保存。
7. 复杂后台管理系统。

## 4. 目标用户

### 4.1 普通旅行用户

用户特征：

- 想快速得到一份旅行计划。
- 不想自己大量查攻略。
- 对预算、交通和行程节奏有一定要求。
- 可能没有明确目的地细节，只知道想去四川。

核心需求：

- 快速生成可参考行程。
- 行程不要太赶。
- 能看到每日安排。
- 能结合预算和个人偏好。

### 4.2 学生和年轻旅行者

用户特征：

- 预算相对有限。
- 偏好美食、自然风景、城市漫步。
- 希望路线轻松、交通简单。

核心需求：

- 明确预算是否够用。
- 减少不必要的交通成本。
- 尽量避免频繁换酒店。
- 获得适合年轻人的美食和景点建议。

### 4.3 项目评审者 / 面试官

用户特征：

- 关注项目是否完整。
- 关注是否只是简单调用大模型。
- 关注后端工程化、RAG、结构化输出等能力。

核心需求：

- 能直观看到 AI 生成结果。
- 能看到 RAG references。
- 能理解系统调用链路。
- 能看到预算分析、异常处理、统一响应等工程化设计。

## 5. 核心使用场景

### 5.1 场景一：生成四川 5 日旅行计划

用户输入：

```json
{
  "departureCity": "重庆",
  "destination": "四川",
  "days": 5,
  "travelers": 2,
  "budget": 6000,
  "preferences": ["美食", "自然风景", "轻松节奏"],
  "avoid": ["太早起床", "每天换酒店"]
}
```

系统输出：

- 旅行计划概述。
- 5 天结构化行程。
- 每天上午、下午、晚上安排。
- 每天交通建议。
- 预算分析。
- 知识库引用来源。

### 5.2 场景二：查看知识库检索结果

用户输入同样的旅行需求，系统只返回知识库检索结果，不调用大模型。

用途：

- 验证 RAG 是否命中正确资料。
- 查看 references 来源。
- 调试关键词检索效果。

### 5.3 场景三：调整条件后重新生成

用户对第一次结果不满意，可以修改：

- 天数。
- 预算。
- 偏好。
- 避免项。

然后重新生成旅行计划。

## 8.2 旅行需求输入区

### 功能说明

用户通过表单输入旅行需求。

### 字段列表

| 字段 | 类型 | 是否必填 | 默认值 | 说明 |
|---|---|---|---|---|
| 出发城市 | 文本输入 | 是 | 重庆 | 用户从哪里出发 |
| 目的地 | 文本输入 | 是 | 四川 | 当前主要支持四川 |
| 旅行天数 | 数字输入 | 是 | 5 | 必须大于等于 1 |
| 出行人数 | 数字输入 | 是 | 2 | 必须大于等于 1 |
| 总预算 | 数字输入 | 是 | 6000 | 单位：元 |
| 偏好 | 多选标签 + 自定义输入 | 否 | 美食、自然风景、轻松节奏 | 用户喜欢的旅行风格 |
| 避免项 | 多选标签 + 自定义输入 | 否 | 太早起床、每天换酒店 | 用户不希望出现的内容 |

### 偏好预设

- 美食。
- 自然风景。
- 轻松节奏。
- 历史文化。
- 城市漫步。
- 摄影。
- 亲子游。
- 博物馆。
- 茶馆。

### 避免项预设

- 太早起床。
- 每天换酒店。
- 行程太赶。
- 高海拔。
- 过度爬山。
- 排队太久。
- 长时间坐车。

### 表单校验

| 字段 | 校验规则 | 错误提示 |
|---|---|---|
| 出发城市 | 不能为空 | 请输入出发城市 |
| 目的地 | 不能为空 | 请输入目的地 |
| 旅行天数 | 大于等于 1 | 旅行天数至少为 1 天 |
| 出行人数 | 大于等于 1 | 出行人数至少为 1 人 |
| 总预算 | 大于等于 0 | 预算不能为负数 |

### 验收标准

- 用户可以完成所有字段输入。
- 用户可以选择和取消偏好标签。
- 用户可以选择和取消避免项标签。
- 表单错误时不能提交。

## 8.3 操作按钮区

### 按钮列表

| 按钮 | 说明 |
|---|---|
| 生成行程 | 调用 `/api/trips/plan` |
| 重置 | 恢复默认输入 |
| 重新生成 | 已生成结果后再次调用接口 |

### 交互规则

- 点击生成时，按钮进入 loading 状态。
- loading 期间禁止重复提交。
- 成功后按钮文案可变为“重新生成”。
- 重置按钮清空当前结果或恢复默认表单。

### 验收标准

- 用户不会因为重复点击导致多次请求。
- 生成中有明确反馈。

## 8.10 错误提示区

### 功能说明

展示接口失败或校验失败的错误信息。

### 错误来源

统一响应结构：

```json
{
  "code": 40001,
  "message": "days: 旅行天数至少为 1 天",
  "data": null
}
```

### 错误类型

| 错误 | 说明 | 前端展示 |
|---|---|---|
| 40001 | 参数校验失败 | 展示后端 message |
| 50201 | AI JSON 解析失败 | AI 返回格式不稳定，请重新生成 |
| 50001 | 系统内部异常 | 系统异常，请稍后重试 |
| 网络错误 | 后端未启动或请求失败 | 请求失败，请检查服务是否启动 |

### 验收标准

- 错误时不清空用户输入。
- 错误提示清晰可见。
- 用户可以修改参数后重新提交。
